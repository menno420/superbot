## Environment Runbook ‚Äî booting & operating the bot in the sandbox

*So a future session knows the possibilities without re-deriving them.*

**Booting the test bot to verify your work is ALWAYS allowed, possible, and safe ‚Äî no
maintainer approval needed.** `DISCORD_BOT_TOKEN_PRODUCTION` is a **separate, dedicated
test bot** ("Galaxy Bot#6724", alone with the maintainer in a private server) ‚Äî it is
**NOT** the real production token and cannot interfere with the live bot, so the only
historical worry (token collision) does not apply. Boot it whenever a live check adds
confidence; do not treat booting as gated.

- **Entry point:** `python3.10 disbot/bot1.py` (Procfile `worker:`). Run from the
  repo root ‚Äî Python puts `disbot/` on `sys.path`, so `import config` etc. resolve.
- **Credentials (already in the container env):**
  - `DISCORD_BOT_TOKEN_PRODUCTION` ‚Äî **misleading name**: it is a **separate, dedicated
    test bot** ("Galaxy Bot#6724", alone with the maintainer in a private server), **not**
    the real production token. Booting it cannot affect the live bot ‚Äî always safe.
  - `DATABASE_URL` ‚Äî points at **localhost**
    (`postgresql://superbot:***@localhost:5432/superbot`), i.e. a *local* DB, not a
    remote prod DB.
  - **Only these two creds are provisioned** (Postgres + the Discord test token), with
    **Full** network access. No AI-provider / YouTube / Paragon / webhook keys are present,
    so those paths run **degraded** ‚Äî see the feature-flags note below.
- **Local Postgres is NOT running by default.** **The cluster is usually already
  initialized** (Debian-packaged: data dir `/var/lib/postgresql/16/main`, config under
  **`/etc/postgresql/16/main/`**, not the data dir). So the fast path is just to
  **start** it ‚Äî don't `initdb` (it errors "directory exists but not empty") and don't
  `pg_ctl -D <data>` (it fails "could not access postgresql.conf" because the conf lives
  in `/etc`, not the data dir). **Working recipe (corrected 2026-06-08):**
  `pg_ctlcluster 16 main start` ‚Üí ensure role+db exist (`DATABASE_URL` =
  `superbot:superbot@localhost:5432/superbot`):
  `su postgres -c "psql -c \"CREATE ROLE superbot LOGIN PASSWORD 'superbot' CREATEDB\""`
  + `su postgres -c "createdb -O superbot superbot"` (skip whichever already exists).
  Only fall back to the raw `initdb`‚Üí`pg_ctl` recipe on a genuinely empty data dir.
  Binaries: `/usr/lib/postgresql/16/bin` (run as the `postgres` user ‚Äî Postgres refuses
  to run as root).
- **Schema bootstraps itself on boot:** `pool.init()` ‚Üí `ensure_migrations_table` ‚Üí
  `create_tables` ‚Üí `run_migrations`. No manual migration step. Migrations
  **auto-discover** by `NNN_<snake>.sql` (no manifest); don't trust a number written
  anywhere ‚Äî check `ls disbot/migrations | tail -1`. (De-numbered 2026-06-10: the
  parenthetical count here rotted twice in two days; the `ls` is the only truth.)
- **Network:** Discord gateway + REST are reachable from the sandbox.
- **One instance only:** a Postgres-backed runtime lock (with heartbeat) lets only
  one bot be active; a second waits. **On restart, kill the old instance first.**
- **Logs:** the bot writes `bot.log` (rotating, repo root) + stdout. Working pattern:
  `nohup python3.10 disbot/bot1.py > /tmp/testbot.log 2>&1 &` then tail
  `/tmp/testbot.log`; put a Monitor on it filtered to `ERROR/CRITICAL/Traceback`.
- **Quality gates (green before pushing):**
  `python3.10 scripts/check_quality.py --full` (CI mirror: black/isort/ruff + mypy +
  the full pytest suite ‚Äî ~8.3k tests, ~95s as of 2026-06-09; the count grows, trust
  the script not this number) and `python3.10 scripts/check_architecture.py --mode strict`
  (0 errors required; pre-existing warnings OK).
- **Env feature flags (OFF here ‚Üí expect *degraded*, not *broken*):** `AI_ENABLED`
  (AI off), `AUTOMATION_SCHEDULER_ENABLED=false` (time/XP auto-assign loop off; manual
  `!assignroles` / "Run Now" works), `YOUTUBE_API_KEY` / `PARAGON_API_KEY`
  (off/degraded), `DISCORD_WEBHOOK_URL` (off). Command access fails **OPEN** on a fresh
  DB (no policy ‚Üí all channels). Owner-gating uses `bot.is_owner` (Discord app owner =
  maintainer); `config.BOT_OWNER_USER_ID` is the AI-side owner identity ‚Äî it also gates AI **scope** (`_derive_scope` ‚Üí `PLATFORM_OWNER`, e.g. the `diagnostics_health_snapshot` tool), not just personalization.

**Production (the maintainer's real bot ‚Äî for diagnosis, never touched from here):**
Railway service `worker`, **auto-deploys `main` on every merge** (restart policy
is on-failure ‚Äî why `!restart` must exit **nonzero**, PR #675). BTD6 data in prod
is the **postgres blob lane** (`BTD6_DATA_BACKEND=postgres`): merged data PRs do
NOT change what prod serves until `!btd6ops seed-data` runs (self-applying since
PR #676; drift surfaced at boot + `!btd6 status`). Full note:
`docs/subsystems/btd6.md` ¬ß Current state. (Learned live 2026-06-10 ‚Äî the
"(55.0)" eval ghost-chase.)

**Boot gotchas (learned the hard way):**

- `pkill -f "disbot/bot1.py"` matches the *launching shell itself* (its command line
  contains that string) ‚Üí it self-kills before relaunching. **Working pattern (used all
  this session):** filter by `comm` ‚Äî it's `python3.10` for the bot but `bash` for your
  shell:
  `for pid in $(pgrep -f "disbot/bot1"); do [ "$(cat /proc/$pid/comm)" = python3.10 ] && kill "$pid"; done`
  then `sleep 3` and relaunch in a **separate** step. Launch via the harness
  `run_in_background` so the bot persists across turns and notifies you only if it exits.
- Running `check_quality`/pytest in the repo leaks test-fixture errors into `bot.log`
  (importing `bot1` installs the file handler). When mining logs, filter by the live
  `boot_id` to separate real bot errors from test pollution.
- The `interaction_router` logs "Unhandled interaction prefix ‚Ä¶" on nearly every panel
  click ‚Äî **benign**: only the `ai` cog registers a router prefix; every other panel is
  handled in-memory by its View. Buttons work; it's log noise.

## Operating Preferences (maintainer)

- Audits live by clicking panels/commands and asking "should X not be Y?" ‚Üí diagnose
  precisely, confirm the root cause, record it; don't wave it off.
- Wants the test bot booted and live logs watched during sessions.
- Root cause over symptom ‚Äî fix the whole class (e.g. central `clamp_embed`, not one
  panel).
- Batch fixes by cog; commit + push each coherent chunk.
- Prefers self-maintaining checks over hardcoded lists.
- A PR at session end is a standing expectation (also in `.claude/CLAUDE.md`).
- Prefers planning prompts to be copy-paste-ready and explicit about hidden
  dependencies, forgotten follow-ups, shipped work that must not be duplicated,
  required tests, rollback safety, and stop conditions.
- **During a live walk the maintainer drives the clicks but wants YOU to lead** ‚Äî give
  numbered, copy-pasteable command steps + "what to look for", not vague guidance ("tell
  me which commands you want me to use step by step"). Findings come rapidly and
  conversationally, usually phrased "X should be Y" / "X is currently Z" ‚Äî treat each as a
  precise diagnosis target: confirm the root cause in source, then fix or log it.
- **Word-cues:** "for now" / "let's forget about that for now" = **defer/drop**, not a
  permanent kill (e.g. the non-reproducible Wallet bug). "as of this PR" = PRs are used as
  state markers. They explicitly want the **central / root-cause fix over per-panel
  patches** ("not 20 patches").

- **‚òÖ Run the exact CI lint commands on changed files right before `git push`** ‚Äî
  `black --check .` / `isort --check-only .` / `ruff check .` with the workflow's
  flags/excludes. PR #528's first `code-quality` run failed `black` (a >88-char line in
  `migrations.py`) + `ruff` D209 (the new `_log_buffer.py` docstring) that weren't
  surfaced at push time. The PostToolUse auto-format hook may not fire in the
  remote/web container, so don't rely on it; verify the formatters yourself.
- **Cog files have an 800-LOC hard ceiling** (`tests/unit/invariants/test_cog_size.py`).
  Adding even ~6 lines to a cog already at the ceiling fails CI ‚Äî `help_cog.py` was at
  exactly 800 this session, which broke the first full run. **Check a cog's LOC before
  adding to it**; wire cross-cutting hooks from `bot1` (the composition root), not from a
  maxed cog. The invariant's intent is to decompose (move view code to `views/<sub>/`),
  not to grow the cog.
- **CI skips the heavy suite on docs-only PRs** (#532): the `code-quality` workflow has an
  in-job changed-files detector ‚Äî docs/`*.md`-only PRs finish in seconds; any `.py` /
  `requirements*` / `.github/workflows/**` change still runs the full suite. A fast green
  on a docs-only PR is expected, not suspicious. (`tests/` is excluded from
  black/isort/ruff/mypy ‚Äî only pytest correctness matters there.)
- **CI installs `requirements.txt` + the pinned tools only ‚Äî NOT `requirements-dev.txt`.**
  (`code-quality.yml` pip-installs black/isort/ruff/mypy/pytest + `-r requirements.txt`, then
  runs `pytest tests/`.) So a **dev-only dependency is absent in CI** ‚Äî e.g. `grimp` (added
  for `scripts/context_map.py`). If a tool/script needs such a dep: import it **lazily** with a
  fallback, and gate any test that needs it behind `pytest.importorskip(...)`, or the suite goes
  red in CI while green locally. Don't "fix" it by moving a non-runtime dep into `requirements.txt`.
- **Real-Postgres test = module-local skip-if-unreachable fixture, never a shared conftest
  one.** CI (`code-quality.yml`) runs **no** Postgres service and `tests/conftest.py` has
  **no** DB fixture (by design ‚Äî every other suite mocks the pool). To exercise real SQL,
  put a `@pytest_asyncio.fixture` *in the test module* that `pytest.skip()`s when
  `DATABASE_URL` is unset (CI) or `pool.init()` raises `OSError`/`asyncpg.PostgresError`
  (DB down), applies schema via `utils.db.pool.init()`, and sweeps a unique fingerprint/key
  prefix on entry+exit. Canonical example: `tests/unit/db/test_health_findings_integration.py`.
  Use before/after **count deltas** (not absolute counts) for whole-table aggregates so a
  booted bot's own rows don't perturb assertions. Pytest runs **serial** here (CI +
  `check_quality`), so a shared module prefix is safe. A test's setup-only raw `UPDATE`/
  `DELETE` is fine ‚Äî the sole-writer AST guards only scan `disbot/`.

### Architecture, data & coding
- **‚òÖ Guard at the mutation seam, not the call site.** A "preflight before mutation" /
  permission / validation check belongs *inside* the service that performs the change
  (e.g. `role_automation.apply`), not in each caller ‚Äî there it protects every current
  and future caller and can't be bypassed by a new one. A dead safety-check wired into
  nothing (`check_preflight` was defined + tested but never called) is worse than none:
  it reads as covered. (Role-automation degradation, 2026-06-06.)
- **`utils.db.pool` has no `fetchval`** ‚Äî use `fetchone` + a `RETURNING` CTE for counts.
- **Expose cog-layer data to services via a `diagnostics_service` provider** ‚Äî cogs
  register INTO services, never the reverse (keeps the services‚Üícogs boundary clean).
- **Pin churn-prone deps** and audit `discord.ui` overrides on version bumps ‚Äî
  unpinned `discord.py>=2.3.0` resolving to 2.7.1 caused `View._refresh` / `Item.parent`
  collisions that crashed the bot.
- **Discord role `position` is NOT unique ‚Äî compare hierarchy by (position, id), never
  raw `position`.** At equal position the *older* role (smaller id) ranks higher, matching
  discord.py's `Role` ordering. Bot-created roles commonly all sit at position 1, so a raw
  `position >=` check mis-flags perfectly manageable roles as "above the bot" (caught live
  2026-06-06). Route every "can the bot manage this role?" check through
  `utils.role_feasibility` (the single source of truth; `_at_or_above` does the tiebreak).
- **Modals CAN contain selects on the current pin** *(corrected 2026-06-12 ‚Äî verified
  against installed discord.py 2.7.1)*: since 2.6, `discord.ui.Label(text=‚Ä¶,
  component=Select(...))` wraps a select inside a modal. The old "modals are text-input
  only ‚Üí restructure modal‚Üíview" advice predates the pin; which select *types* Discord
  accepts in modals is probe-pending (UX Lab plan P-07/P-08,
  `docs/planning/ux-lab-interface-gallery-plan-2026-06-12.md`). Existing modal‚Üíview
  restructures remain fine ‚Äî just don't cite the old rule to block a Label-based design.
- **Discord caps bite on every catalogue growth** ‚Äî re-check all four whenever content
  tables grow: **25 options/select** (a silent `options[:25]` slice = items invisible ‚Äî
  the market panel had exactly this latent), **1024 chars/embed field** (clamp_embed
  truncates silently), **5 rows/view**, **100 chars/select-option description**. The
  V-16 gear catalogue (15‚Üí41 shop items, 18‚Üí44 recipes) tripped two at once
  (2026-06-11, PR #702).
- **Validate a derivation against ground truth before shipping** ‚Äî a migration-table
  regex silently missed runtime-created tables.
- **A test can encode a bug ‚Äî read what it asserts, not just whether it's green.** When a
  write convention and a read convention must agree, pin the agreement (a live round-trip),
  not one side's current value. (`scope_id == 0` was a passing test that locked in a silent
  no-op.)
- **Grep ALL references (incl. tests) before a rename** ‚Äî a missed test mock cost a red
  suite mid-session.
- **Invoking a Discord view button/select callback in a test:** discord.py 2.7's
  `@discord.ui.button` / `@discord.ui.select` decorator **shadows the instance attribute
  with the `Button`/`Select` component** ‚Äî `panel.my_btn` is the *component*, not the
  coroutine, so `await panel.my_btn(interaction, btn)` raises `'Button' object is not
  callable`. The original coroutine stays on the **class**: call
  `type(panel).my_btn(panel, interaction, btn)` (or `PanelClass.my_btn(panel, ‚Ä¶)`). A
  `Modal.on_submit` is a plain method ‚Äî call it directly. (Probed live 2026-06-08 building
  P0C's Seed-Defaults test; no folio covered "how to drive a view callback in a unit test".)
- **The role-threshold XP-cache-invalidator invariant hides in `test_xp_cog_caching.py`**
  (`test_threshold_role_mutation_sites_import_invalidator`): it requires
  `views.roles.{time_roles_panel,xp_roles_panel,creation_panel}` + `cogs.role_cog` to each
  **import `invalidate_xp_threshold_roles`**. When you route a threshold write through the
  audited `role_automation` seam (which invalidates internally), **keep the module's import**
  or that invariant reddens CI ‚Äî and its filename doesn't say "roles", so grep the invariant
  set rather than assume what constrains a change. (P0C, 2026-06-08.)
- **Even a *fresh* same-day planning note needs source-verification before you build on it.**
  The 2026-06-08 plan's ¬ß16.8 item 2 said the P1B drift providers should "run
  `resolve_feature_access` against the live guild" ‚Äî but the projection short-circuits on the
  first deny, so it can't expose an *axis disagreement* (routing vs command-access); you read
  the policy owners directly. Corrected in ¬ß16.8 item 6. The "verify cross-agent output vs
  source" rule applies to your own prior session's notes too, not just Codex/ChatGPT.

### Cross-agent & git workflow
- **‚òÖ-candidate ‚Äî Derive the clean/durable answer; don't surface it as an owner
  question when the documented principles already settle it (2026-06-13, Q-0119).**
  When a decision reduces to "the clean / durable / root-cause / one-source-of-truth
  fix vs. cheaper-but-inconsistent alternatives," the working agreement *already
  determines* it ‚Äî **decide the clean option and record it as a _derived_ decision**
  (with the rationale + which principle drove it), rather than handing the owner a
  multiple-choice. **"Architectural" is not itself an ask-trigger**; ask only when the
  principles genuinely don't disambiguate (a real product / taste / cost fork, or
  something irreversible). This also covers an *inherited* OPEN router Q a prior
  session over-deferred ‚Äî close it by derivation. (Miss: surfaced Q-0119's (a)/(b)/(c)
  where (a) was both the documented clean home *and* my own recommendation; owner
  flagged it should have been derived. CLAUDE.md is already explicit ‚Äî "if you're about
  to offer options you expect rejected, you've answered your own question ‚Äî act.")
- **‚òÖ-candidate ‚Äî With native auto-merge armed, land ALL commits (code AND docs)
  before CI goes green (2026-06-13, #794‚Üí#797).** `auto-merge-enabler` arms native
  auto-merge at PR-open and GitHub merges the instant `code-quality` passes ‚Äî
  server-side, ~15s. Docs / session-log commits pushed *after* the first green strand
  on the branch post-merge and need a recovery follow-up PR (#794 merged at its first
  head before the docs landed ‚Üí had to open #797). So either **bundle docs into the
  same push as code**, or run the doc-freshness + session-log step *before* the first
  push. (Successor to the Q-0093 "merge in-turn" lesson, which predates native
  auto-merge.)
- **Never stamp a PR number you haven't created (3 mis-stamps on 2026-06-10:**
  **#673‚Üíok, #677‚Üí#678, #680‚Üí#682).** Parallel sessions consume numbers, so a

  `create_pull_request` ‚Üí THEN write the number into docs/logs (the Q-0052
  draft-first convention exists exactly for this; it covers ready-PRs too).
- **‚òÖ Before starting ANY implementation slice, check live GitHub for an open PR
  on the same item (2026-06-10, the #677/#678 collision).** Two sessions built
  plan-PR-A of the Help editor simultaneously ‚Äî the queue names items but nothing
  claims them; the loser cost a full duplicate build. `list_pull_requests`
  (state=open) is one call; an open PR titled into your plan = pick a different
  queue item or continue THAT branch. This is Q-0060's named revisit trigger
  (first accidental same-item recurrence ‚Äî a second one justifies the
  active-sessions ledger option). **(Extended 2026-06-11:) also check what
  MERGED to main since your orientation docs were written** ‚Äî `git fetch origin
  main` + skim `git log <base>..origin/main --oneline` for same-area commits.
  PR #701 (the gear sprite seed, merged minutes before the gear session began)
  prepared assets *for* that session but appeared in no orientation doc; the
  session discovered it only at final-merge sync and had to reconcile a
