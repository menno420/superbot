## Architecture review: superbot-next (`sb/`, commit 443617a) vs old superbot (`disbot/`, FETCH_HEAD 1df81f8c)

### 1. OLD — how the three subsystems actually look in `disbot/`

**Settings.** Not as pathological as its reputation — a June refactor already built an audited `SettingsMutationPipeline` (`disbot/services/settings_mutation.py`, 686 lines, 11-step contract), and the draft lane delegates to it (`setup_operations.py:1284` `_apply_set_setting` calls `SettingsMutationPipeline().set_value`). **But scatter survives**: raw un-audited `db.set_setting` writes remain in `services/btd6_version_announce.py:88-97`, `services/btd6_ct_team_service.py:66,72`, `services/tournament_state_service.py:79,88`, `services/ai_review_log_service.py:427`, and `governance/writes.py:568` — plus a raw `db.get().execute(...DELETE FROM subsystem_visibility...)` at `governance/writes.py:559` violating the repo's own "never call execute outside utils/db" rule. So settings writes go through **≥3 distinct paths**.

**Moderation/warn.** Also already centralized at the service seam (`services/moderation_service.py`, 743 lines; `warn` at :402 owns the escalation ladder "the cog + the seven modals used to copy"). Rendering is per-surface string building in the cog (`moderation_cog.py:166-175`) and views (`views/moderation/modals.py:83,142`).

**Blackjack.** Genuinely scattered: settle/payout logic lives in the **view** (`views/blackjack/solo_view.py:224-237` payout table; `economy_service.credit`/`debit` called from view code at :73/:81) *and* duplicated in the **cog** (`blackjack_cog.py:492-496` — instant-blackjack path recomputes `int(bet*1.5)` and credits economy directly from a cog). Money writes from views and cogs, not a service.

**Tracked debt:** `architecture_rules/layers.yaml` carries **55 `known_violations` entries** plus a `known_lazy_violations` class; **617 `discord.Embed(` call sites** across `disbot/`; total 243,544 LOC / 881 py files.

### 2. NEW — is the centralization real?

Largely **yes, structurally**:

- **One settings read seam**: `sb/kernel/settings/__init__.py:150` `resolve()` — undeclared key ⇒ `LookupError("no raw-KV reads")`; tri-state chain + activation terminus; declarations minted only via `register_manifest_settings` (:190). **One write lane**: 4 K7 `CompoundOpSpec`s in `sb/domain/settings/ops.py`; there is exactly **one** `INSERT INTO settings` statement in the whole repo (`sb/kernel/db/settings.py:162`), and `_declared_or_refuse` (`ops.py:53`) refuses undeclared keys at the write leg.
- **One render path**: exactly **1** `discord.Embed(` site in `sb/` (`sb/adapters/discord/panel_view.py:43`) vs 617 in old. Panels are declarative `PanelSpec`s rendered by a discord-free kernel (`sb/kernel/panels/render.py`), budgets/custom_ids/nav enforced once.
- **One mutation spine**: `sb/kernel/workflow/engine.py` — authority leg, idempotency (`once()`/`record_outcome`), DB legs + central audit in one txn, outbox emits, post-commit EFFECT legs with compensators, and a structural dry-run oracle (txn rollback + skip-effects). All moderation, settings, and blackjack money movement route through it; blackjack credits go through `wager.credit_in_txn` inside the op txn (`sb/domain/blackjack/ops.py:120-147`) instead of view/cog-level `economy_service.credit`.
- **Layer purity is enforced, not exhorted**: `sb/domain/` contains **zero** `import discord` (only `sb/adapters/` touches discord), and the manifest compiler bans `"import discord"`/`".emit("` tokens inside DB legs (`sb/kernel/workflow/compile.py:50`). 28 checker tools in `tools/`. Handlers are genuinely thin (e.g. `blackjack.play` at `sb/domain/blackjack/handlers.py:62-94` just parses a mention and calls `engine.run`).

**But it is not all real:**

- `sb/domain/settings/ops.py:8` claims "these four ops are the **only** write paths to the settings table" — false: `sb/domain/settings/service.py:192-204` `_state_write` fire-and-forgets a direct `db_settings.upsert_setting` (no K7 op, no audit, swallowed exceptions) for platform-governance latches.
- Blackjack tournament orchestration is **stubbed**: `sb/domain/blackjack/handlers.py:118-126` registers `pending_handler`s ("needs the live orchestration … arms with the live adapter at CUT-1"); old-bot crash-recovery sweeps (`blackjack_cog.py:157-300`) have no visible successor yet.
- Moderation target-DM notification is **dropped**: old `_notify_target` (`moderation_service.py:280`) fires on every warn/action; new `dm_member` exists only as a port protocol method (`sb/domain/moderation/service.py:169`) that **nothing calls** — the parity transport even logs it as a `gap` (`sb/adapters/parity/transport.py:343`).

### 3. TESTS (sampled ~12 of 1,025 collected `def test_` functions; claim was ~1077 — close, parametrization plausibly covers the gap)

Behavioral majority, at the leg/seam level with substantive fakes:

- `tests/unit/workflow/test_engine.py` (32 tests) — genuinely behavioral: `test_required_leg_failure_rolls_back_and_returns` (:197) asserts rollback + no audit rows; `test_durable_once_replay_reproduces_without_reeffect` (:151); `test_effect_leg_compensation_fork_e` (:249) asserts compensator ran + PARTIAL; `test_preview_dry_run_oracle_byte_identical_zero_effects` (:293). The `FakeConn` (`conftest.py`) actually simulates idempotency-key uniqueness and audit SQL — not mocks-echoing-mocks.
- `tests/unit/band6/test_band6_blackjack_rps.py` — behavioral money conservation: `test_solo_stand_settles_and_clears` (:92) asserts `balance == 100 + after["delta"]` and checkpoint consumed; `test_pvp_stand_both_settles_pot_only`, overdraft refusal, double-down funding. Fakes in `tests/unit/band6/conftest.py` implement real wallet/checkpoint contracts.
- `tests/unit/band2/test_band2_slice1.py` — `test_warn_escalation_ladder` (:80) asserts the 3-row history vocabulary and verbatim ack copy; `test_timeout_record_leg_parses_duration` (:126).
- Structural minority: `test_manifest_compiles_in_isolation`, `test_ops_register_and_are_natural_key`, `test_legacy_vocabulary_is_complete_and_verbatim` (`tests/unit/settings_band/test_band1_settings.py:150-205`).

**Would they catch a real logic regression?** *Within next*, yes — the settle table, ladder rows, tri-state resolution, and engine txn semantics are all pinned. **Relative to the old bot, no** — see the warn case below: the test *enshrines* the changed semantics. The instrument that would catch old-vs-new drift is the 465-golden parity harness (`parity/` — real captured transcripts with message content + full DB deltas, e.g. `parity/goldens/blackjack/blackjack_solo_round_hit.json`), but `parity/parity.yml` shows **all 50 subsystems `pending`, zero `ported`** — the golden gate's report leg is "0/465, red-until-parity by design" (`.github/workflows/golden-parity.yml:13`). The oracle exists and is excellent; it is not yet gating anything.

### 4. Duplication check

**Old N places → new 1 (proven):** the blackjack payout/credit. Old: `views/blackjack/solo_view.py:73` (`new_bal = await economy_service.credit(...)` inside a view's `_finish`) **and** `cogs/blackjack_cog.py:492-496` (`payout = int(bet * 1.5) ...; new_bal = await economy_service.credit(...)` inside the cog command) — money movement in two UI layers. New: every credit/debit for solo, PvP escrow, and tournament flows through `wager.credit_in_txn`/`debit_floor_in_txn` inside the K7 transaction in one module (`sb/domain/blackjack/ops.py` `_settle_solo` :120), audit rows included. Same for settings: 3 write paths → 1 op lane (one `INSERT INTO settings` in the repo).

**Counter-case (proven):** the new bot re-created per-module boilerplate duplication the reports call centralized — the identical frozen `class Reply: outcome/user_message` dataclass is copy-pasted **23 times** and the identical `_ctx_from_req(req, params)` helper **16 times** across `sb/domain/*/handlers.py|service.py` (karma, xp, rps, economy, blackjack, mining, chain, …). It's the old "duplicated helpers per cog" pathology in new syntax, just smaller. Plus the `_state_write` settings-table bypass contradicting `ops.py`'s sole-writer docstring.

### VERDICT

**(a) Mixed — genuinely-improved architecture, incompletely-proven behavior.** The centralization is real and enforced (one write engine, one render path, one settings seam, discord-free domain layer, compile-time fences) — this is not the old structure renamed; handlers verifiably delegate rather than re-implement. But the port has stubs (tournament orchestration), dropped features (moderation DM notify), a sole-writer violation (`_state_write`), fresh boilerplate duplication (Reply/`_ctx_from_req` ×16-23), and — decisively — none of the 465 behavioral goldens gate it yet.

**(b) Biggest provable win:** the K7 workflow engine (`sb/kernel/workflow/engine.py`) — every mutation in all three sampled subsystems now passes through one transaction + audit + idempotency + dry-run + compensation spine, behaviorally tested (rollback, replay-dedup, compensation), replacing the old bot's 55 tracked layer violations and view/cog-level DB writes. The 617→1 `discord.Embed` collapse is the most visually dramatic corollary.

**(c) Biggest provable risk:** port-time semantic drift that the new unit tests enshrine rather than catch. Concrete case: old warn (`disbot/services/moderation_service.py:453-473`) does **not** reset the warning count and reports `escalation_blocked=True` when Discord refuses the escalation; new warn (`sb/domain/moderation/ops.py:52-96`) resets the count and writes the escalation + "Warnings cleared" history rows **inside the DB txn, committed before** the post-commit EFFECT leg (engine step 5, `engine.py:311-315`) ever attempts the Discord action, with no compensator on WARN (`ops.py:332-334`) — a blocked escalation now yields a phantom "timeout/kick/ban" mod-history row and a wiped count. `test_warn_escalation_ladder` (`tests/unit/band2/test_band2_slice1.py:80`) asserts exactly this new behavior ("count reset IN the leg"). With parity at 0/50 subsystems ported, this class of drift is currently invisible to CI.

**(d) Confidence internal logic quality exceeds the old bot's: 6/10** — the invariant discipline, single seams, and engine-level tests are clearly superior engineering, but until the golden-parity gate flips subsystems to `ported`, correctness-vs-shipped-behavior rests on docstring "verbatim" claims, and I found one core moderation flow where that claim is already false.

**Summary:** superbot-next's centralization is architecturally real (1 write engine, 1 render path, 1 settings seam vs the old bot's 55 tracked violations and 617 embed sites) with mostly behavioral tests, but it re-duplicates handler boilerplate 16-23×, stubs/drops some shipped behavior, has a provable warn-escalation semantics regression its own tests enshrine, and its 465-golden acceptance oracle currently gates zero subsystems — mixed verdict, 6/10 confidence it exceeds the old bot's logic quality.

### 2. The counterfactual

**Defect classes by correlate:**

- **(i) Worker context limits** — surprisingly little direct damage. The 4KB brief cap pushed real instructions into repo files (retrospective §6), which *improved* durability (the band-3 kill was recovered "entirely from durable state", §3). The real context-shaped cost is **seam blindness**: each band worker owned a vertical slice; the horizontal invariant "declared panels actually register / commands actually route" belonged to no one until CUT-1. Items 2–3 above are seam defects. A single deep session holding all bands in one context would very likely have noticed the first `LookupError` while writing band 1 — *if* it ever booted; the plan deliberately had no `main()` until #54.
- **(ii) Spec-without-oracle building** — the dominant avoidable class (items 4–7). 49 PRs merged against unit fakes and checkers, 0 live boots, 0/465 parity flips, parity report red-by-design. The subtle failure: **red-by-design destroyed the per-defect signal.** Band-2 replay was "0/15 pre- AND post-fix" — a dashboard that is red either way cannot tell you `!timeout` ignores its duration; only a human classifying each red (which is what the testing session then did, expensively, in D-0050/D-0052) recovers the signal. The goldens were an oracle sitting on disk the whole time; they were used as a *ratchet* (A-16) but not as a *build-time test*.
- **(iii) Merge-train races** — real but cheap: #44/#46 (gate leak, ~6 min repair), websites #19/#24 (same class), #63 conflict (~30 min), #35 frozen (owner fix), "behind" churn under the up-to-date ruleset (flag 4; same phenomenon old superbot measured as 2h28m merge tails on 2026-07-08). Note also the merge window itself: ~40 of the 49 build PRs merged **7–60 seconds** after opening (e.g. #32: created 23:48:35Z, merged 23:48:42Z). Checks ran push-side before PR-open, so this doesn't prove no gating — but it does mean the PR stage was provenance, not review; nothing human or independent ever sat between a worker's green and main.

**Caught by the fleet's own gates (credit where due):** #54's Postgres bug (live-test doctrine, first boot); PR #40's "real, not flaky-red" CI failures (checker fleet, pre-merge); force-flip pressure resisted twice (D-0050: "no exemption rows minted — the pre-approved exemption kinds do not describe missing subsystems"; A-16 one-way door held at 0/465); the #51 doc's own strict-findings (#53); the band-3 crash (progress-log discipline); owner answers "applied and ledgered in PR #34 rather than acted on informally" (§3).

**Where a single dedicated deep session wins:** items 2–5 likely caught at build time (one context, one boot, one end-to-end drive); no wake noise, no relay hops, no provisioning deaths, no brief caps. **Where it loses:** 14 hours exceeds any single context — old superbot's *own journal* documents what long sessions do under fatigue: forgotten merges (#778), flip-before-scan (#1728→#1730), wrong-branch merges from a masked exit code, guessed PR numbers (3 mis-stamps in one day). A crash mid-run loses the session; the fleet's band-3 kill cost ~one worker respawn. And the eval log independently corroborates the retrospective's net: "overhead is real but linear — it does not compound as the build grows, whereas a single long session degrades."

**The correct counterfactual isn't fleet-vs-deep-session — it's build-all-then-test-all vs build-and-verify-interleaved.** The same 18 workers, with two standing obligations per band — (1) boot the composition root and drive one command live (the flag-5 recommendation the fleet itself wrote: "live-boot smoke per band is the cheap catch"), (2) replay the band's own goldens and classify every red or fix it — would have converted items 2–7 from day-2 rework into day-1 build steps, at a cost of maybe 20–30 min per band. The fleet converged on exactly this doctrine *after* first contact; it just wasn't in the loop during the run.

**Are those signatures in today's output?** Mostly **no** — and in several cases the new repos show the *inverse*:

2. **Red-by-design masking** — an honest-pending dashboard that carries no per-defect signal until a human classifies the reds (D-0050/D-0052 taxonomy work).
3. **Silent sessions / no fleet view** — eval log 14:52Z: "with 10 Projects running, the owner has no single view of which agents are running… sessions actively running workers looked dormant"; plus the manager's fix-and-merge worker denied by the auto-mode classifier (`[Self-Approval]`), and heartbeat-PR staleness (#60).
4. **Test-environment hygiene** — the old prod bot sharing `!` on the test guild "contaminated the owner's session" (flag 15): doubled acks and error scrollback consumed a chunk of the owner-feedback triage for nothing.

### VERDICT

**(a) Quality sacrificed vs a deliberate approach: REAL but localized — not diffuse.** Two concrete places: (1) the **human-facing layer** — the fleet shipped 37 subsystems without a single human-shaped verification, and the entire first owner contact (9 observations, an owner-ordered rework wave, 5 PRs: #63/#65/#67/#70/#71) is the bill for that; (2) **functional gaps that the on-disk oracle would have caught** — silent success, ignored timeout duration, a confirm gate with "no path forward on ANY surface" all merged as "done" because band goldens were ratcheted, not replayed, at build time. Everything else — the Postgres bug, the composition-seam gaps — is inherent or pre-ledgered discovery any approach pays on first live contact. Critically, the *old* problems (drift, overclaiming, duplication, false greens) did **not** come back; the day's problems are new-shaped and were fixed forward within hours, each with a named class preventing re-diagnosis in bands 3–9.

**(b) Gates that actually paid for themselves today (evidence):** live-test doctrine (#54's unit-fake-invisible bug on the first-ever boot; the owner's own keystrokes as evidence in the 12-tail/step-3 rows); **golden pending discipline / A-16** (0/465 held honestly under pressure twice — "no exemption rows minted"; independently verified truthful); **durable-state handoffs + decision ledger** (band-3 kill recovered cleanly; #804-class duplicate work absent all day); **checker fleet** (PR #40 pre-merge, #51's own findings); the **friction→guard loop itself** (born-red gate leaked → websites #24 + kit v1.0.0 fixed it upstream the same day — the gate class earned its keep by failing loudly once, not silently forever).

**(c) The 2–3 process changes with the best quality-per-speed ratio:**
1. **Walking skeleton + live-drive per band (make flag 5 binding):** the composition root is PR #2-class, not PR #54-class; every band worker boots and drives one of its own commands through the real pipeline before merge. Converts the whole #56/#58/#62/#63/#65 class into build-time catches at ~20 min/band.
2. **Classify-or-fix reds at build time:** a band worker replays its own goldens and must sort every red into a named ledger class or fix it in the same PR — keeps the honest-pending posture while restoring the per-defect signal that bulk red-by-design destroyed.
3. **A human-eye gate for user-facing surfaces:** every PR shipping a rendered surface attaches the rendered output (embed dump/screenshot) and a one-line "a human can complete this flow" assertion — the one defect class (presentation, dead buttons, silent sheds) that all 22 checkers plus the parity harness were structurally blind to. Secondary: enforce control-PR terminal states (a heartbeat PR that isn't merged/closed same session is a red status line — #60 is open right now).

**Summary:** The fleet's speed traded away build-time use of its own oracles (goldens, live boot, human eyes) — costing one same-day rework wave in the presentation/UX layer and ~6 fix PRs — but reintroduced none of the old bot's drift/overclaim/duplication signatures, and its gates (live-test doctrine, A-16 pending discipline, durable handoffs) demonstrably caught what they were designed to catch; the highest-leverage fixes are per-band walking-skeleton live-drives, classify-or-fix golden replays, and a human-eye gate on rendered surfaces.
