# Websites fleet data plane — centralized ingestion + misplaced-file failsafe

> **Status:** `plan` — design brief for the **websites lane** (execution routes there as a
> manager ORDER; this hub doc is the owner-reviewable design). **Provenance:** owner ask,
> owner-live hub chat 2026-07-13 (PR #2066): *"find out how all the websites get their live
> data, and if it's possible to make it better and centralized, and includes a failsafe to
> prevent confusion between files from multiple repos — if an agent misplaces a file but
> correctly names it, the website will know how to find it anyway, and based on the
> timestamp it knows which one is most current."* Evidence basis: read-only survey of
> `menno420/websites` @ HEAD 2026-07-13 (all file:line cites below are that repo unless
> prefixed) + superbot's producer side. Verdict up front: **possible, incremental, and most
> of the primitives already exist** — this design consolidates them and adds the resolver.

## 1 · How the sites get their data TODAY (grounded map)

Four Railway services, four different data styles:

| Service | Style | Detail |
|---|---|---|
| **control-plane** (`app/`) | almost all **request-time** | GitHub REST + raw fetches per page: `/fleet` + `/freshness` parse fleet-manager's `scripts/gen_roster.py` **script internals** (the `LANES` literal, `app/fleet.py:581-633`) + every lane's `control/status.md`; `/prompts*` + `/projects/*` fetch the fm registry live (`app/prompts.py`, `app/prompt_history.py`); `/orders`/`/queue`/`/reviews`/`/ideas`/`/journal` fetch per-repo control files; Railway GraphQL for env pages. In-memory TTL cache 180s, negative-caching for stable 40x only (`app/github.py:201-207`). |
| **botsite** | mixed | superbot's committed **contracted feed** `botsite/data/site.json` fetched raw at request time (`botsite/data_source.py:28-31`); hand-committed JSON pages; `graveyard.json` baked from trading-strategy. |
| **dashboard** | **contracted feeds** | superbot's `dashboard/data/dashboard.json` + `botsite/data/console.json`, fetched raw at request time with **pinned schema-version contracts** (`dashboard/data_source.py:36-43,157-232`; producer-side CI enforcement in superbot — the BUG-0022 desync killer). |
| **review** | fully **baked** | network-free at runtime; committed `review/data/*.json` produced by the daily `review-bake.yml` cron (first scheduled success 2026-07-13, run 29235587736). |

**Producer side (superbot):** `scripts/export_dashboard_data.py` is the single producer of
`dashboard.json` / `site.json` / `console.json`, committed in-repo on the dashboard-refresh
loop, with versioned shape contracts (`dashboard/data/dashboard_data_contract.json`,
`botsite/data/console_data_contract.json`).

**Verified pain points (from the websites code itself):**
- **Five separate fleet-repo lists** in the websites repo alone (`app/roster.py:17-33` ·
  `app/config.py:97-182` · `app/config.py:211-338` FLEET_LANES ("kept in sync by hand …
  drift-risk flagged") · `review/gen_fleet.py:108+` · the `app/prompts.py` pinned artifact
  list) — plus the primary live source being *another repo's Python literal*. The repo's own
  story doc: *"hand-kept lists keep drifting against globbable truth — the same failure
  shape found three times"* (`review/story.py:744`); *"a generated lanes.json has been
  requested but not yet shipped."*
- **Three near-duplicate HTTP/TTL fetch layers** (`app/github.py`, `botsite/data_source.py`,
  `dashboard/data_source.py`) + two urllib copies in bake generators; five UA strings;
  `listfilter.py` vendored ×3.
- `control/status.md` **hardcoded in 4+ places**; tokenless deploys sit on the anonymous
  60 req/h API ceiling (`review/story.py:375`).
- No discovery: a 404 at the canonical path is a rendered error, never a search.

## 2 · The design — one manifest, one fetch core, three lanes, a resolver

### 2.1 One manifest, derived — never hand-kept

- **fleet-manager publishes `registry/lanes.json`** as a *contracted feed* (generated by the
  same Actions regen that builds the roster — the feed websites already asked for). Kills
  the script-literal parsing and all five hand lists.
- **Each repo already self-declares its artifact homes** in `substrate.config.json`
  (`heartbeat_files`, `sessions_dir`, `claims_dir`, `docs_root`, `state_dir` — verified
  present fleet-wide). The data plane reads the declaration instead of hardcoding
  `control/status.md`.
- A small **artifact-kind table** (in the data-plane module) maps each kind → name pattern +
  canonical home + parser: `status` → `control/status*.md` · `order/inbox` →
  `control/inbox.md` · `verdict` → `sims/**/VERDICT*` / `verdict-###` · `session card` →
  `<sessions_dir>/YYYY-MM-DD-*.md` · contracted feeds → their contract files.

**Composition:** repo set = `lanes.json` × per-repo `substrate.config.json` × kind table.
Adding a seat or moving a file home becomes a data change, not a websites code change.

### 2.2 One fetch core

A single shared module (vendored to the four services exactly the way `listfilter.py`
already is) owning: raw-first fetch with contents-API fallback (`app/github.py:334-368` is
the donor), one UA, one TTL/negative-cache policy, bounded error envelopes (#237→#244
chain preserved), optional token. The three duplicate layers + two urllib copies collapse
into it. `review` stays network-free at runtime — its *bake generators* adopt the core.

### 2.3 Three lanes by freshness need (keep them — don't force one style)

- **LIVE** (request-time, small files): heartbeats, inboxes, prompts — as today, through
  the core.
- **BAKE** (scheduled, expensive): git-history data, per-repo **tree scans**, stats —
  extends the proven `review-bake.yml`.
- **CONTRACTED FEEDS** (producer-committed JSON, pinned schema): superbot's pattern,
  generalized — producers publish feeds; websites never parse another repo's internals.

### 2.4 The resolver — the owner's failsafe, hardened

Per artifact kind, per repo:

1. **Canonical path first** (from the manifest). Hit → done; no scanning on the fast path.
2. **Miss (404) or stale beyond the kind's threshold → name-pattern discovery**: scan the
   repo tree for the kind's name pattern (one recursive Git-Trees call per repo in the
   LIVE fallback, cached; full scan routinely in BAKE).
3. **Recency = git commit time, NOT the in-file stamp.** This is the design's main
   correction to the sketch: `updated:` stamps are agent-written and have a *proven*
   failure class (the malformed games stamp fixed by ORDER 037 on 2026-07-13; stale and
   wall-clock-faked stamps are documented fleet incidents). The file's last-commit
   timestamp is ground truth an agent cannot mistype. In-file stamps stay as display +
   cross-check — **a material stamp/commit-time disagreement is itself a flagged finding**,
   never silently resolved.
4. **Precedence: canonical wins; newer-but-misplaced never silently replaces it.** If
   discovery finds a same-kind file that is strictly newer by commit time, the page renders
   the canonical data **plus a loud drift row** ("newer `status.md` found at `<path>`,
   committed `<time>` — possibly misplaced"), and if only the misplaced copy exists, it is
   rendered **labeled `discovered`**, never passed off as canonical.
5. **Self-healing, not just tolerant:** the BAKE sweep emits a machine-readable
   `misplaced.json` (repo · kind · found-path · canonical-path · commit-time · sha). The
   fleet-manager's wake reads it and routes a fix ORDER to the owning seat. A site that
   silently forgives misplaced files trains agents to keep misplacing them — the failsafe
   must keep the site truthful *and* make the error visible.
6. **Cross-repo confusion:** discovery is scoped to the kind's *owning* lane (one writer
   per file, fleet doctrine). A same-kind file surfacing in a non-owning repo (a sim-lab
   verdict inside idea-engine) is **flag-only** in `misplaced.json` — never rendered as
   truth.
7. **Provenance on every rendered datum:** (repo · path · sha · commit-time ·
   `canonical|discovered`) — the existing `/version` + drift-row culture, extended to data.

### 2.5 Companion measure — prevent at the source (superbot/kit side)

The website-side failsafe catches misplacement at *render* time; the cheaper catch is at
*commit* time: a kit gate check (`check_misplaced_artifacts`) that scans the committing
repo for artifact-kind files outside their `substrate.config.json`-declared homes and
warns in the substrate gate. Routed separately to the kit lane (idea filed in this PR's
session card). The two halves make misplacement a self-correcting event end-to-end.

## 3 · Feasibility & cost

**Yes — every mechanic already exists somewhere in the codebase:** trees/contents listing +
envelope classification (`app/github.py:103-159`), TTL caches, the bake lane, pinned
contracts, per-repo self-declared manifests. The work is consolidation plus the resolver
(~the same shape as shipped nights' work in this lane). Rate limits: BAKE does the heavy
scans; the LIVE fallback is one cached call. The B#49 read-PAT (already queued as
OQ-FM-ROSTER-READ-PAT) would lift the anonymous 60 req/h ceiling for the same clicks.

## 4 · Build order (routable as websites ORDERs)

- **P1** — fm publishes contracted `registry/lanes.json` (roster-regen rides along); websites
  `app/` swaps script-parsing → feed; one fetch core lands in `app/` (no behavior change).
- **P2** — manifest resolver (lanes.json × substrate.config.json × kind table) adopted by
  `/fleet`, `/freshness`, `/orders`; delete the five hand lists.
- **P3** — the failsafe: 404/stale → discovery + commit-time recency + drift rows +
  `misplaced.json` in BAKE; fm wake consumes it.
- **P4** — botsite/dashboard adopt the core; review's bake generators adopt it; kit-side
  `check_misplaced_artifacts` (kit lane, separate).

Each phase is independently shippable and reversible; P1/P2 remove the drift class the
websites repo has hit three times before the failsafe even lands.
