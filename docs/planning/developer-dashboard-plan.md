## Architecture

### Two zones (hard boundary)

* **Public zone** (no auth): showcase, function/command catalogues, public bug-report form.
* **Private zone** (owner auth — Phase 2): checklist, ideas/bug management, the AI control board,
  and the **secrets** area.

The secrets area is **never** reachable from the public side.

### Stack & decoupling

FastAPI + Jinja2 + Tailwind (CDN, no JS build step), Python 3.10+, with its own small Postgres for
mutable state (Phase 2: checklist, bug mirror). `dashboard/` **does not import `disbot/`** — it reads
the generated `dashboard/data/dashboard.json` and (later phases) the GitHub + Railway APIs.

### Data flow

`scripts/export_dashboard_data.py` (pure stdlib) reads the repo sources and writes
`dashboard/data/dashboard.json`; the app renders that JSON. Sub-scanners (also stdlib):
`scripts/scan_env_usage.py` (env map) and `scripts/scan_commands.py` (cogs/commands). The committed
JSON is the deployed data — **regenerate + commit after source changes** (or wire a CI/cron refresh — see ideas).

## Operating the service via the Railway API (reuse for Phase 3b)

The service was created and is operated entirely through the Railway GraphQL API (no CLI). Phase 3b
(managing the bot's env vars from the dashboard) reuses this exact path, so the method is recorded:

- **Endpoint:** `https://backboard.railway.com/graphql/v2`
- **Auth:** `Authorization: Bearer $RAILWAY_API_KEY` — an Account-scoped token present in the agent
  environment (value never committed). Account scope = can read/write all services in the project.
- **Cloudflare gotcha:** send a **browser `User-Agent`** header. Python's default `python-urllib/*`
  UA is blocked (HTTP 403, body `error code: 1010`). `curl` works by default; `urllib` needs the header.
- **Mutations used this session:** `serviceCreate` · `serviceInstanceUpdate` (`rootDirectory`,
  `startCommand`, `healthcheckPath`) · `serviceConnect` (`{repo, branch}`) · `serviceDomainCreate` ·
  `serviceDomainUpdate` (needs `serviceDomainId` + `serviceId` + `environmentId` + `domain`) ·
  `serviceInstanceDeployV2(serviceId, environmentId)`.
- **Status/logs:** `service(id){ deployments(first:N){ edges{ node{ id status createdAt } } } }`;
  build/runtime logs via `buildLogs(deploymentId)` / `deploymentLogs(deploymentId)`.

## Secrets — done safely (Phase 3)

1. **No second vault.** Secrets stay in Railway (encrypted); the dashboard is a *UI over Railway's
   env vars* via the API — no second copy to breach.
2. **Usage map ✅ shipped** (`/env`, `scripts/scan_env_usage.py`): names + code locations only, never a value.
3. **Value management (Phase 3b, not built):** masked view/edit behind owner login, audited, public
   zone has zero secret access. Uses the Railway API above.
