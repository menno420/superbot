# Fleet-wide repo audit as a saved, reusable workflow (2026-07-13)

> **Status:** `ideas` — session idea (Q-0089, from the EAP-final-night fleet cleanup audit
> session). Not approved for implementation.

## The idea

Tonight's owner-directed fleet cleanup audit (`docs/eap/fleet-cleanup-audit-2026-07-13.md`)
proved a pattern worth encoding once: **one subagent per repo, each carrying (a) shared
global safety rules — never touch a draft / do-not-automerge / owner-sweep-only PR or
anything touched in the last 2-3h; only merge freshly-verified green+clean work; never
blind-resolve a real code conflict; never self-merge its own report PR — and (b) a
repo-specific liveness hint distilled from the freshest fleet-manager worklist/roster —
audits its repo, safely disposes of what it can, and returns a structured report** that
rolls up into one consolidated cross-repo finding doc.

Across 18 sibling repos, this pattern merged/closed **zero** PRs inappropriately (every
"hands-off" call held, even against 11 repos where a live coordinator was mid-flight and
the worklist hadn't caught up yet) while still surfacing 10 real cross-cutting patterns
(worklist staleness, dead `project.index.json` scaffolding, control-bus size walls, a
GitHub MCP tooling quirk, and more — see the report). Encode it once as a saved workflow
(`.claude/workflows/fleet-repo-audit.js`, parameterized by: the repo list, per-repo
liveness hints pulled live from `fleet-manager`'s roster/worklist at run time instead of
hand-pasted, and the report-file convention) so a future periodic sweep — end-of-EAP-window
style, or a standing monthly/weekly cadence — runs by `Workflow({name: 'fleet-repo-audit',
args: {...}})` instead of re-authoring the ~200-line script from scratch.

## Why it's worth having

- The safety envelope is the hard-won part: getting "never touch a draft/owner-gated/recent
  PR" right across 18 heterogeneous repos with wildly different cadences (one merge every
  2 minutes in idea-engine vs. wind-down-complete in the codetool-labs) took real iteration
  this session, including one caught-and-corrected instance where an early draft of the
  script let agents self-merge their own report PR (blocked by the harness's own permission
  classifier, not by the script's own design — worth hardening the template so that mistake
  can't recur).
- A generic version could pull its per-repo liveness hints live from `fleet-manager`'s
  generated `docs/roster.md` at workflow-start instead of a human hand-summarizing it into
  the script — closing the exact "worklist staleness" gap the audit itself flagged as the
  #1 cross-cutting finding.
- Cheap to run repeatedly: 18 agents, ~2.2M subagent tokens, ~59 minutes wall-clock for the
  full fleet tonight (per the workflow's own usage report) — proportionate for a periodic
  hygiene pass, not a standing per-session cost.

## Guardrails to carry over verbatim

- Every merge/close decision re-verifies live state immediately before acting (never trusts
  a stale doc) — this is what let every repo correctly self-correct when its own "0 open
  PRs" briefing turned out to be minutes stale.
- Agents never self-merge their own report PR — flagged and fixed once this session; make
  it structurally impossible in the saved version rather than relying on prompt wording.
- Agents should not be granted `subscribe_pr_activity` unless standing PR monitoring is
  actually the ask — several subagents reached for it unprompted tonight, creating
  session-scoped webhook noise beyond the bounded one-shot audit (see report §5).

## Open questions for whoever picks this up

- Should the per-repo activity hints be generated by a first pipeline stage (read
  fleet-manager's roster + worklist live) before the audit stage fans out, rather than
  hand-authored? That would remove the one manual-curation step left in tonight's version.
- Where should the consolidated cross-repo report land by default — always superbot's
  `docs/eap/`, or configurable per invocation?
