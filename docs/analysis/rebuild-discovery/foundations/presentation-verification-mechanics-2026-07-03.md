## 1. Executive summary

### State of the SURFACE + PROVING foundation

The honest headline is that **most of the presentation substrate already ships and is stronger than the plan assumes, but almost none of the *proving* half exists, and the plan's flagship "one engine / one grammar / one config" claims are targets, not shipped state.** Across all 46 mechanics the same shape recurs: a working imperative implementation exists (single `HelpCategoryView` + `HelpProjection`; `CardCanvas`/`Theme` engine with 5-6 consumers; `PersistentView` + `@register` restart re-registration; per-domain result dataclasses), but the *declarative grammar* the rebuild is named after — `CardTemplateSpec`, `WorkflowResult` rendering, the one-description manifest block, the "template = named draft over a HubConfig" primitive — is undesigned, and the *oracle* that would prove any of it correct is absent in every mechanic. This is the biggest structural finding: **the foundation is an engine-rich, grammar-thin, oracle-empty layer.** Six of the ten highest-severity issues are either `thin-step` (a named-but-undesigned grammar) or `verification-hole` (no oracle for a feature with no parity golden).

The single most foundational defect is **restart-safe navigation**. The decided contract (Q-0231 "Back = pop the real path you took") has no serializable storage medium: today's Back is a non-persistable in-memory `BackTarget`/`chain_back` closure that `navigation.py` *explicitly forbids* persisting, so under the merge=deploy cadence every redeploy collapses the back-trail to the semantic parent. This one gap blocks hub-topology, the navigation engine, panel-rendering, and the re-homed preset views simultaneously, and it has no oracle. It should be the owner's and Session A's first joint look, because "pop the real path" must be pinned to a restart-safe medium (versioned custom_id via `DynamicItem`, or a DB panel-state row) *before* Gate-0 or the whole navigation contract is decorative.

The second foundational cluster is **fragmentation that the plan under-sizes**. Presets/templates are reimplemented ~13-15 times across two semantically distinct families (draft-bundle vs policy-value), the "one config, two views" unification between the setup preset selector and the help overlay editor does not exist (they are structurally disjoint stores), per-command descriptions have three divergent authoring lanes, card rendering has 3 off-engine renderers duplicating the PIL path, and outward result rendering is 667 raw ✅/❌ literals with denial written 12+ ways. Each is a real collapse opportunity, but each is larger than one plan line implies and none has the acceptance oracle (preview-fidelity golden, one-edit-propagates test, result-matrix golden) that would keep the collapse honest.

**What the owner should look at first, in order:** (1) the restart-safe Back-path storage decision (blocks four mechanics); (2) the **hide-vs-disable contract collision** — the plan's recommended "hidden = off, not runnable" default directly *reverses* the shipped, drift-tested Q-0055/HLP-4 invariant that display-hide is presentation-only, and it surfaces in three mechanics, so it must be resolved before any hub/preset spec freezes; (3) the admin/moderation gating model (the shipped bot *hides* admin from non-admins, the opposite of the decided "gated visible node," and a single admin gate strips moderators of their door). These three are contract-level and gate the rest.

## 2. Cross-mechanic themes

The findings cluster into six recurring patterns. Each is a *class* of gap that shows up across several mechanics — the highest-leverage places to fix once rather than per-subsystem.

### T1. Restart-safety under merge=deploy is unsolved at the framework level

The decided navigation/persistence contracts assume state survives redeploys, but the shipped mechanisms that would carry it don't: Back is a non-serializable in-memory closure navigation.py explicitly forbids persisting; custom_ids are static/unversioned; anchored panels re-attach interactivity but never re-render content; re-homed preset/help views still use 180/300s per-instance timeouts. Every 'survives restart' claim is optimistic until a serializable medium (versioned custom_id via DynamicItem or a DB panel-state row) and a cross-deploy sim exist.

*Touches:* `navigation-engine, hub-topology, panel-rendering, interface-presets-template`

### T2. Named engine, undesigned grammar (the thin-step pattern)

The rebuild is named after declarative data grammars — CardTemplateSpec, the WorkflowResult render surface, the one-description manifest block — but in each case only an imperative engine ships and the layout/outcome/description grammar itself is entirely undesigned. The collapse targets (667 ✅/❌ literals, 140-line pixel-math card functions, three description lanes) have nothing to collapse *into* until the grammar is specified.

*Touches:* `card-engine, result-grammar, help-projection, media-generation`

### T3. The proving half is missing — every engine, no oracle

Almost every mechanic names an engine and omits the oracle that proves it. No test asserts Back+Home hold on every state/depth/re-render/restart; no preview-fidelity golden asserts preset preview == live hub; no result-matrix golden over (outcome × render_mode × lane); no fake-provider harness proves the media cost/abuse posture (at-cap = zero calls, cache-hit = zero calls, kill-switch stops next call); no one-edit-propagates test for the unified description; no manifest-completeness checker. These features have no parity golden, so an automated oracle is mandatory, not optional.

*Touches:* `navigation-engine, card-engine, media-generation, interface-presets-template, help-projection, result-grammar`

### T4. Fragmentation the plan under-sizes — collapse onto one seam

The same 'many implementations, one intended seam' pattern recurs and the plan's counts undercount it: presets/templates ~13-15x across two families (plan says ≥7); card renderers 3x off-engine with 3 copies of pillow_available(); per-command descriptions 3 authoring lanes plus a reflection catalog; result rendering 667 literals + 12+ denial variants + 6-way color; two disjoint customization stores. Each collapse is real but larger and needs its own acceptance oracle.

*Touches:* `interface-presets-template, card-engine, help-projection, result-grammar, hub-topology`

### T5. Hide-vs-disable contract collision (display-hide ≠ execution-off)

The plan's recommended 'hidden = off, not runnable' default directly reverses the shipped, drift-tested Q-0055/HLP-4 invariant that hiding is presentation-only and never disables execution. This single unresolved contract question surfaces as a distinct issue in three mechanics and gates whether Back/Home may target a hidden node and whether a preset can disable vs merely hide — it must be resolved by the owner before any hub/preset/nav spec freezes.

*Touches:* `hub-topology, interface-presets-template, navigation-engine`

### T6. Refusal / denial / error are forgotten states, not first-class panels

The outward grammar models success but not the states that fragment worst today: the WorkflowResult vocabulary omits user-authority DENIED and unexpected ERRORED (the 12+ denial variants and the crash path can't flow through the one renderer); media refusal/degraded states (quota, budget cap, kill-switch off, safety refusal) and the 'generating…' lifecycle aren't modeled as navigable panels; nav-target resolution doesn't re-check authority to rehome away from a now-hidden node. Denial/error/refusal should be first-class navigable outcomes through the same renderer.

*Touches:* `result-grammar, media-generation, hub-topology, navigation-engine`

## 4. Owner-gated decisions (flagged, not decided)

These are the calls the audit **surfaces for the owner** — contract-level or cost/privacy choices the agents must not decide. Ordered roughly by how many mechanics they gate. (The first three are contract-level and block spec-freezes downstream — see the executive summary.)

1. Hide-vs-disable contract: does display-hide EVER mean execution-off? The plan's 'hidden = off' default reverses the shipped, drift-tested Q-0055/HLP-4 'hiding is presentation-only' invariant — rule for the whole hub/preset/nav layer before any spec freezes.
2. Admin/moderation gating model: adopt the decided 'gated visible node' (show-locked) vs the shipped 'hidden from non-admins', declare a per-hub hide-vs-lock policy, and make the gate multi-tier (moderator + administrator), not a single admin flag.
3. Deep-link command naming: reconcile shipped entry_commands (!adminmenu/!modmenu/!economymenu, two names for the admin door) against Q-0231 §2's decided !admin/!games — pick the canonical set.
4. Two customization stores: approve collapsing the setup-preset store (SERVER_PRESETS→SetupOperation drafts) and the help-overlay store (help_overlay rows) onto one per-guild HubConfig, and budget the reinvention.
5. Media global budget cap: decide the dollar/spend cap policy and that a spend counter (which does not exist today) is built as the highest-risk money control.
6. Media PII scrub: decide whether member display names/usernames must be scrubbed from generation prompts (numeric IDs already are), beyond the generic redaction seam.
7. Media safety posture: ratify fail-CLOSED (refuse on any budget/safety/provider fault) as one inherited rule, reversing the current fail-OPEN moderation default for a payer-controlled generator.
8. Media cache semantics: decide whether prompt-hash cache-hits make 'regenerate' return the identical image, and whether cached images may be reused cross-guild (privacy scope).
9. Native-Discord interop: decide whether C-3 presets interoperate with, or replace, Discord Community Onboarding / Server Templates before shipping a competing onboarding system.
10. C-7 description scope: decide whether the per-guild overlay's rename/re-describe lane extends down to individual commands, and whether a single prose description is expected to seed the fuzzy candidate set and NL intent surface.
11. Result-grammar ephemerality: decide the home for the silent-vs-reply / ephemeral-vs-public policy so the 1084 scattered ephemeral= call sites collapse to one lane-driven resolver.
12. Card engine customization: decide guild-customizable themes/backgrounds (server-uploaded images) and the color-emoji asset font, which the plan currently strips/hardcodes.
13. 'Did you mean' privacy: rule what 'private' means for a prefix-command typo (the rung yields a message, not an interaction, so true ephemeral is impossible) — the recommended 'invoker-locked public carrier + ephemeral followup' vs a DM.
14. Panel custom_id standard: ratify 'static namespaced ids are stable/unversioned; versioning is reserved for the dynamic session scheme' and reconcile the Q-0231 'versioned custom_id' wording accordingly.

### 6.1 Hub topology — one unified help hub with admin as a permission-gated node (Q-0230)

**How it works now.** The bot ALREADY ships a single help front door — not two player/operator trees — but admin is HIDDEN from non-admins, not shown-locked, which is the opposite of the decided Q-0230 model. `!help`/`/help` opens ONE `HelpCategoryView`: a dropdown of mother hubs (games, btd6, project_moon, economy, moderation, community, utility, admin) whose rows come from `HubEntry` registry entries filtered through ONE `HelpProjection` visibility seam. Admin ('Server & Admin', minimum_tier=administrator) already lives INSIDE that one hub as a tier-gated entry, and click-time governance re-check already exists (hub_children callback) — so much of the Q-0230 substrate is present. What is missing: `_hub_decision` maps a tier-floor failure to GOVERNANCE_HIDDEN (a hiding state), so a non-admin never sees an admin door at all; hubs have no 'advertised-but-locked' state. Moderation is a SECOND privileged tier (minimum_tier=moderator), so 'one admin gate' is already a simplification. Navigation is best-effort auto-attach: a '📚 Help' button doubles as Home (absolute), and '↩ Back to <parent>' is a SEMANTIC-parent builder, NOT a real path-stack pop; BackTarget explicitly cannot survive restart, and nav is silently dropped at the 25-component cap. Presets/customization exist TWICE: the setup preset selector writes SetupOperation drafts from automation_templates.SERVER_PRESETS, while the help-overlay editor writes separate help_overlay rows — two stores, and the current preset 'preview' lists staged operations, not a rendered hub.

**Adversarial-verify correction (Q-0120).** The finder's how_now is accurate against shipped source — every cited file:line checks out. Verified: ONE help front door (help_cog.py:326-327 builds a single HelpCategoryView, invoke-and-see with prior-anchor delete); the Home dropdown is built from projection.visible_hubs() (panels.py:74); admin is a single administrator-tier hub entry, entry_command='!adminmenu' (hub_registry.py:238-253) and moderation is a SECOND privileged tier, minimum_tier='moderator', entry_command='!modmenu' (hub_registry.py:163-184); tier-floor failure maps to GOVERNANCE_HIDDEN not a locked state (help_projection.py:456-463) and _HIDING_STATES only ever hides (help_projection.py:101-103, with the confirming comment at :98-100 "Help intentionally shows locked features … lock states never hide"); click-time governance recheck exists (hub_children.py:160-170); BackTarget MUST NOT be persisted across restart (navigation.py:134-139); Home=NAV_HELP_ID "nav:help", Back=semantic-parent "↩ <hub>", and there is NO NAV_HOME id and no path stack (navigation.py:58-59, 407-456); the 25-component cap returns False and only logs "User cannot return from this panel" (navigation.py:200-208); the preset preview is an ops-list of op.kind lines, not a rendered hub (preset_select.py:97-116); the help-overlay editor writes via help_overlay_mutation and carries the Q-0055 "hidden from Help but still executable" semantic (editor.py:34-40, 52). Two clarifications, neither weakening the finder: (a) the admin door has TWO shipped names — the slash command is `admin` (admin_cog.py:51-52) while the hub registry entry_command rendered in Help is `!adminmenu` (hub_registry.py:243, printed at help_cog.py:240) — which strengthens the naming-handoff point; (b) the Q-0232 §3 "hidden = off" recommendation is real and explicitly still-open in the decision doc (rebuild-hub-navigation-presets-2026-07-03.md:89-91,119), and it does contradict the shipped Q-0055 contract. ## 9. Appendix — method, coverage, caveats

**Method (the workflow).** Discover phase: one finder agent per mechanic — (a) find-how-now against shipped source with `file:line`, (b) research 2–3 alternatives (discord.py, ProBot, MEE6, Carl-bot, Dank Memer, pycord + UX research), (c) pressure-test our decided approach — each finding then **adversarially re-verified** by an independent agent that re-opened the cited files and killed/《REVISED》 any claim that fought the source (Q-0120). Completeness phase: three critic lenses (competitor / verification / lifecycle) proposed forgotten in-scope mechanics; each genuinely-new one ran the same finder→verify pipeline; the loop ran to its round cap. Synthesis phase: ranked the ledger and extracted themes + owner-gated flags.

**Scale.** 108 subagents · 195 CONFIRMED + 25 REVISED findings (REJECTED dropped) · ~9.0M tokens · 2,276 tool calls.

**Coverage caveats (honest limits — no silent truncation):**

- **2 mechanics were lost** to a `StructuredOutput` retry-cap failure in the workflow (a production *observability/SLO* contract finder and one other) — their finders errored after 5 invalid-schema attempts, so they carry no entry here. The observability/SLO gap (presentation SLOs: ACK-budget breach rate, render-failure rate, nav-path traversal telemetry) is real and should be picked up in the Stage-2 walk.
- **Near-duplicate mechanics.** The completeness loop's weak dedup surfaced several overlapping oracle/lifecycle mechanics (e.g. `live-view-timeout` ≈ `graceful-component-expiry`; the static vs dynamic navigation-conformance oracles). They are kept distinct here for traceability; Stage-2 should **merge** them when specifying — the overlap is itself a signal that these contracts are one family.
- **One finder emitted placeholder output** (`test-mechanic`); the adversarial-verify pass caught it and reconstructed the real *mobile-viewport legibility* analysis from source — the entry shown is the verified reconstruction, not the placeholder.
- **Source wins (Q-0120).** Line numbers are as-of this session's HEAD; re-read before acting on any single citation.

**What this feeds.** The survivors + owner-gated flags feed the next planning stage (Stage-2 subsystem walk / Gate-V verification fleet, Q-0234). The **oracle fleet** (§7 E) is the concrete build-list for the rebuild's *proving* half — the class-8 verification-hole answers. Owner-gated items ([§4](#4-owner-gated-decisions-flagged-not-decided)) are the gate before any hub/preset/nav/result spec freezes.
