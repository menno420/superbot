## Purpose

This report is a **docs-only discovery and audit** of the runtime/logic foundational mechanics that the SuperBot rebuild uses now or could use — the "engine room" beneath every feature, the layer that decides *what happens* upstream of how it is shown. It is **PROMPT A** of the two-session foundational-mechanics brainstorm authorized by owner ruling **Q-0236** (PROMPT B, session B, covers the presentation/interaction half). Its job is to **surface and flag, not decide**: for each mechanic it lays out how the system works today, what leading bots and frameworks do differently, and where the current or frozen approach carries risk, and it explicitly **flags the calls that are owner-gated** rather than making them. Nothing here touches runtime code or settles a disposition — every recommendation is an option or a flagged issue for owner review, and the deliverable is the rubric-scored issues ledger, not a verdict.

## Method

Each mechanic was worked in three passes, then rolled up.

- **Per-mechanic pass** — (a) *how-now*: the current implementation, cited to `file:line` in shipped source; (b) *alternatives*: two or three approaches drawn from leading bots and frameworks (the discord.py / Discord ecosystem and general service-runtime practice); (c) *pressure-test*: a critical read of the frozen or decided approach against those alternatives and against known failure modes.
- **Adversarial-verify pass** — every *how-now* claim was re-checked against shipped source under **Q-0120**: a claim that fights the evidence is killed, source wins on facts, and nothing survives on assertion alone.
- **Completeness-critic loop (3 rounds)** — repeated "what mechanic or failure mode is still missing?", each round feeding new entries back into the discovery set; the 35 mechanics indexed below are the closure of that loop.
- **Synthesis** — findings were folded into a single **rubric-scored issues ledger**, ranked so the owner reviews the highest-leverage calls first.

## Scope & boundary

This report's scope is the **engine room**: the runtime and logic substrate — manifest grammar / compiler / snapshot, the namespace and command registries, the four-rung invocation ladder and the resolver they converge on, the authority / audited-mutation / composition pipelines, the event bus, lifecycle and persistence, concurrency and locking, the DB and data seam, and the operational contracts (migrations, pool lifecycle, health, config, scheduling, durable delivery, sharding). It deliberately **stops at the session-B boundary**, which owns presentation and interaction: hub / navigation, panel rendering, the card + media engine, interface presets, help/description projection, the *rendering* of the response/result grammar, the *rendering* of "did you mean", the critical-review rubric itself, the correctness oracle, and the layout-success simulator. Several mechanics span both halves — e.g. the response/result grammar (logic decides the result; session B renders it) and the fuzzy / "did you mean" path (logic scores the match; session B renders the suggestion). **Rule: where a mechanic spans both sides, this report covers only the logic side and marks the handoff seam** so session B can pick it up cleanly.

## Frozen dispositions NOT re-litigated

The **2026-07-03 owner decisions (Q-0219 through Q-0236)** and the frozen **BUILD-PLAN / FINAL-REVIEW dispositions** are treated as **settled inputs, not open questions**. This report *pressure-tests* them — it may show where a frozen choice is under tension, or surface a **new** issue that choice creates downstream — but it does **not re-open or re-argue the decision itself**. Consistent with **Q-0120**, source and recorded owner rulings win on facts: a pressure-test that merely disagrees with a frozen call is out of scope; a pressure-test that produces a concrete, evidenced *new* failure mode is a flagged issue for the ledger.

## Mechanics inventory

### Manifest grammar + compiler + snapshot
**How now.** The current bot ships a *projection-based* manifest spine (Q-0162), not a declaration→runtime pipeline — the arrow runs backward. Config IS declared: `SubsystemSchema` (`subsystem_schema.py:234`) declares settings/bindings/resources/domain_panels only — it has no commands/panels/events/tasks/stores fields (`subsystem_schema.py:270-276`) — and `settings_registry.build_registry()` (`settings_registry.py:181`) projects it. Commands/panels are *reflected from the hand-wired runtime*: `command_manifest` is built from the live `bot.commands` ledger walk (`command_manifest.py:5-7,246`) and `panel_manifest` by instantiating+introspecting registered `PersistentView`s (`panel_manifest.py:162`). `manifest_reconciliation.reconcile()` (`manifest_reconciliation.py:66`) cross-checks them but *defers* button→command binding (`:22-27`). There is **no compiler** — startup is sequential non-fatal glue (`bot1.py:1180-1249`) — and **no on-disk snapshot**: "snapshot" means an in-memory frozen dataclass (`command_manifest.py:113`); `tools/manifest_compile.py` and `*.snapshot.json` are absent. The six amendments G-1..G-6 are durable but spread across four docs (`spec.py` markers, `RESULTS.md:14`, `rebuild-linchpin-validation-2026-07-02.md:185`, folded into design-spec §2 `:41-44`).
**Options considered.** discord.py/Pycord decorator-registration + CommandTree remote-sync (no declaration/compiler; = today, plus the hash-diff sync at `bot1.py:305`); Red-DiscordBot `Config` registry (settings-as-declaration only, no unified manifest); poise/serenity/hikari command-as-data struct trees (data builds the *command* runtime, but no committed snapshot, no S/A/O roles, no simulator, no generated help/router/docs). None generates the whole surface — UI/help/router/import — from one snapshot; that unification is the rebuild's differentiator and its novelty-risk.
**Recommendation.** **Keep the decided §2 direction — it root-fixes the shipped drift class and is measured (73%→85%, 97% operator band).** But three load-bearing pieces are unbuilt and under-specified and must land at K2 with real spec, not prose: (1) the compiler's validation passes (§2.0 has ~1 paragraph); (2) the manifest→discord.py *generation* step + an inverse "runtime matches manifest" kernel gate (mirror of `test_command_manifest.py:57`) wired to the existing slash-sync diff — this is the literal "is there a real compiler" gap; (3) the amendment registry as sole minting authority (idea `rebuild-amendment-registry-2026-07-03.md`, filed-not-built) before folding "the Gate-0 amendments". Maps to Q-0219 (S-1, incl. guardrail-7 anti-inner-platform for `gate`/`visible_when` predicates), Q-0222 (selective-import data oracle), and I-1.
**Handoff seam (session B).** Reuse `spec.py`→`sb/spec/` and the 3 spike manifests verbatim; before writing the compiler, enumerate its passes and build the store-completeness reconciliation that distinguishes D-5 "intentionally not ported" from a forgotten StoreSpec (owner-gated — silent prod-data loss). Carry the working `command_tree_sync` hash-diff (`bot1.py:305`) into the snapshot→remote parity story, which §2 currently omits.

### Namespace registry — collision prevention (K1)
**How now.** No pre-boot registry and no runtime command-name reservation exist. A second claimant of a top-level name/alias makes discord.py raise `CommandRegistrationError` at `add_cog` (`disbot/bot1.py:704-720`; confirmed `.sessions/2026-06-29-railway-give-collision-crash.md:14`); the failed cog's missing `entry_points` (`bot1.py:722-729`) trip the STRICT identity-contract, which aborts the whole boot (`bot1.py:1104-1170`, `subsystem_registry.py:1603`) → crash loop → offline. The `command_surface_ledger` `duplicate_command_names` finding walks `bot.commands` only *after* all cogs load, so it is structurally blind to the collision that stopped one from loading (`command_surface_ledger.py:607-625,213-226`). Real prevention is three **reactive, prefix-only** static guards (`tests/unit/invariants/test_extension_integrity.py:89,138,165`; `test_cog_load_smoke.py:87`), each added after an outage. The only existing reservation in the running bot is for capability-string prefixes, not commands (`subsystem_registry.py:1244-1250`).
**Options considered.** (a) discord.py native — runtime `CommandRegistrationError` at registration (what crashes SuperBot today; no pre-boot signal). (b) Red-DiscordBot — per-cog isolated load, bot stays up on a failed cog (SuperBot's own degrade-don't-die idea). (c) Carl-bot/Dyno/MEE6 — reserved built-ins + per-guild `(guild_id,name)` uniqueness, user custom commands validated against the reserved set *at set-time* (the runtime-reservation pattern K1 lacks). (d) serenity/Pycord — declare-up-front, duplicates caught at build/sync but no cross-kind registry.
**Recommendation.** **Change (build K1) — strongly.** It kills the exact class (Q-0211 `give`, BUG-0030 `dock`/`sail`) by moving detection design-time/CI/pre-boot and generalizing to all name kinds (C-5/C-1, Q-0224, `design-spec:102-104,294-298`). Keep the kind-scoped partition (G-6, `design-spec:549`). But close four source-shown gaps before the freeze: bind the registry to a **runtime `is_reserved()` query** for Q-0225 custom triggers; scope reservation by **(kind, parent-group)** not flat; make the **CI and pre-boot checks share one oracle** and retain the boot smoke test; fold the `give` ban + tombstones into the registry (retire the `BANNED_COMMAND_TOKENS` hardcode) and drift-check `legacy_reservations.json`.
**Handoff seam (session B).** K1 kernel owns the engine; resolve the runtime-trigger query, group-scope granularity, CI↔pre-boot oracle equivalence, and ban/tombstone-as-reservation before any domain declares. Stage-2 input: compute + publish the shared-verb reservation set over the 271-command corpus.

### Command naming — shared-verb rule (Q-0224)
**How now.** No design-time naming rule or namespace registry exists — K1 is explicitly "the one thing that doesn't exist today" (`docs/analysis/rebuild-discovery/new-bot-capability-audit/findings/NEW-BOT-BUILD-PLAN.md:137`). One global prefix `!` (`disbot/config.py:28`) + slash. Grouping is ad hoc in three coexisting styles: concatenated pseudo-namespaces (`@commands.group(name="btd6ops")` → `@btd6ops.command(name="readiness")`, `disbot/cogs/btd6_ops_cog.py:41-52`), real discord.py groups (`ticket close`, `karma add`), and bare flat commands. Separators diverge by kind — prefix concatenates/snake_cases (`adminmenu`, `source_enable`) while slash hyphenates (`bot-audit-log`, `why-no-response`) (grepped `disbot/cogs/`). Collision handling is **runtime**: `command_surface_ledger` reports `duplicate_command_names`/`duplicate_alias_names` as post-hoc findings (`disbot/core/runtime/command_surface_ledger.py:607-625`), with no live gate (`tests/unit/runtime/test_command_surface_ledger.py:261-284`) — the hard stop is discord.py raising at registration. A shipped precedent for the decided approach exists: the BTD6 unification collapsed 5 prefix groups into one `/btd6` tree "so users no longer have to remember which prefix owns which action," keeping old prefixes as hidden aliases (`disbot/cogs/btd6/_unified.py:1-40`). Measured cross-cog verb sharing over live source: **11 verbs across ≥2 cogs** (`add`×4, `list`×4, `status`×4, `remove`×3, …); over the frozen `ground-truth/command-surface.json` the same computation yields only **1**, because that 271-row corpus stores flat leaf names only (btd6_ops = 1 group row for 5 real subcommands).
**Options considered.** Red-DiscordBot: hybrid commands + human-declared groups with a `fallback` subcommand (native parity, but grouping is judgment → the drift SuperBot is escaping). Carl/Dyno/MEE6: flat curated verbs + web dashboard (max discoverability, but only scales at tens, not 271; flat-only blows Discord's 100-top-level cap). Dank Memer: configurable invocation string as data (orthogonal — complements Q-0225 triggers, doesn't solve verb collision). discord.py `app_commands.Group`: the substrate whose caps (100 top-level / 25 sub / 1 nest / lowercase `^[\w-]{1,32}$`) are the mechanical oracle the computation must budget against.
**Recommendation.** **Keep** the decision (it correctly generalizes the BTD6 pain into one mechanical rule and beats both human-judgment grouping and flat-only) but **tighten the executable spec before Stage-2 runs it**: (i) compute the shared-verb set from the live ledger/manifest qualified-name walk, not the incomplete JSON (maps to C-7 one-description-surface + the K1 reservation input); (ii) pin *area* = `subsystem_registry` key; (iii) budget against Discord's 100/25 caps (owner fork: slash-vs-prefix-only cut); (iv) resolve the bare-shared-verb entry point via the Q-0225 fuzzy rung / discord.py `fallback`; (v) add a declared `effect` field so the safe-no-arg-default is a mechanical oracle, shared with Q-0226 mod-actions-as-data. All feed K1 (Q-0224 reservation set) and Gate-0.
**Handoff seam (session B).** Before the Stage-2 subsystem walk publishes per-subsystem flat-vs-grouped lists, freeze the shared-verb set from `command_surface_ledger.build_ledger()` (`disbot/core/runtime/command_surface_ledger.py:336-386`) — **not** `ground-truth/command-surface.json` — bundling area-definition, `/area`-token map, cap-fit budget, and the `effect` field into K1's reservation input.

### Invocation rung 1 — exact (slash / prefix / additive custom triggers)
**How now.** Prefix is a single global static string — `command_prefix=config.PREFIX` (`bot1.py:81`), `PREFIX="!"` (`config.py:28`), `case_insensitive=True` (`bot1.py:93`); there is no per-guild/callable prefix (grep finds no callable/list form). Prefix commands dispatch via discord.py's built-in `process_commands`, while the app's platform `on_message` is a *separate* listener owned by the message pipeline (`message_pipeline.py:355`). Slash exists but is a minority, separately-coded surface: 225 `@commands.command` + 19 groups vs 31 `app_commands.command`, **0 hybrid** (e.g. `moderation_cog.py:77` prefix `modmenu` + `:92` slash `modmenu`, but `ban`/`kick`/`timeout` prefix-only at `:207/:181/:155`). Prefix and slash have two separate admission gates (`bootstrap_access_cog.py:135` and `:298`). **Additive custom triggers do not exist**: no custom-command/trigger DB table (migrations `000-103` have none), aliases are static in-code (`treasury_cog.py:44`), and keyword detectors (`four_twenty_cog.py:135`, `utils/projmoon/keywords.py:23`) are hard-coded and observe-only, never invoking a command. On a non-matching prefix input the bot is **not** silent today — `CommandNotFound` runs the fuzzy resolver and replies with auto-correct / 'Did you mean?' / '❌ Command not found. Use !help' (`bot1.py:541-586`), a path with a documented infinite-loop history (BUG-0014, `bot1.py:547-560`).
**Options considered.** Carl-bot: up to 15 *replacement* per-server prefixes + custom-command/tag store (rich but replace-prefix is a lockout footgun; parallel invocation surface). Dank Memer: fixed bareword `pls` (the prior art, but non-configurable, un-scoped). Red-DiscordBot / discord.py: Alias cog (additive guild aliases, un-scoped, un-validated) + callable `command_prefix` for per-guild prefixes + `hybrid_command` to declare prefix+slash from one definition (the canonical fix for the 0-hybrid duplication).
**Recommendation.** **Keep + centralize + extend.** Keep slash-default + prefix; **centralize** all front-ends onto one manifest-declared command via **C-1 / S-1 (Q-0219)** to erase the 225/31/0-hybrid split and the twin admission gates; **extend** with the additive scoped triggers (genuinely new), whose union model + set-time validation are correctly chosen over Carl-bot's replace model (**Q-0225 §2.1**). Ship four things this decided approach currently leaves thin: (a) place trigger-resolution as one ordered pipeline stage *after* auto-mod (extend the `message_pipeline.py:44-58` table); (b) define the rung-1-miss → rung-2-fuzzy fall-through; (c) a property test for the additive-never-disables invariant; (d) slash **command groups** to live under Discord's ~100-global cap vs ~244 commands.
**Handoff seam (session B).** This rung is the foundation for rungs 2-4 (S-2 / Q-0220): land it first, exposing `resolve_exact(scope_ctx, token) -> CommandRef | None` (additive union of global/guild/channel/user) as a single ordered pipeline stage; session B's fuzzy resolver plugs in as the declared fall-through when it returns None (do not rebuild fuzzy inside the error handler as today, `bot1.py:541`). Set-time validator + invariant test ship with this rung.

### Invocation rung 2 — the fuzzy typo matcher (LOGIC)
**How now.** A central command-typo resolver **already exists** and already ships the decided three tiers: `disbot/utils/command_resolution.py` exposes `Outcome{AUTO, SUGGEST, NONE}` (`command_resolution.py:74-79`) = run-directly / did-you-mean / silent, driven by `difflib.SequenceMatcher` (`:88-89`) at `cutoff=0.8` (`:61`), with a `_MIN_AUTO_LEN=4` gate (`:63-66`), an `_AMBIGUITY_DELTA=0.08` downgrade-to-SUGGEST guard (`:71,162-169`), and a hand-maintained `DESTRUCTIVE_COMMANDS` frozenset that is never auto-run (`:38-57`). `derive_auto_correct_set` double-gates AUTO on min-length, non-destructive, and lexical isolation (`:92-121`). It is wired in `disbot/bot1.py` from a `token->canonical` map of live names + aliases + the hand-kept synonym table (`bot1.py:451-469`), cached on command-surface size (`:472-479`), and fires only on prefix `CommandNotFound` (`:541-544`) with a BUG-0014 loop-breaker before re-dispatch (`:547-572`). SUGGEST currently renders a **public** `delete_after` message (`:577-581`). An exact-alias layer feeds it (`utils/synonyms.py:6-69`). The genuinely scattered difflib **domain-value** resolvers are two, each with its own cutoff: `utils/mining/names.py:44-67` (`cutoff=0.7`) and `cogs/counting/parsing.py:143-148` (`cutoff=0.8`); BTD6 uses a third, non-difflib substring style (`btd6_data_service.py:1526-1640`). **The decision doc's premise is wrong (Q-0120: source wins):** the named anchors do **not** use difflib — `ai_preset_service.py:21` is exact-only, `channel_recommender.py:163-168` is keyword scoring with confidence buckets, `setup_ai_advisor.py:340-370` is an LLM confidence field.
**Options considered.** discord.py raw — no built-in matcher, everyone hand-rolls difflib in `on_command_error`; slash avoids typos via autocomplete (general knowledge). Red-DiscordBot — a first-class fuzzy "did you mean" menu gated by a setting, but suggest-only, no auto-run (general knowledge). Dank Memer/MEE6/Carl/Dyno — slash-first + silent-ignore on unknown prefix, never auto-execute a guess; modern libs prefer rapidfuzz over difflib (general knowledge).
**Recommendation.** **Centralize — but as a *port + generalize* of the existing `command_resolution.py`, not a greenfield build (C-5, §2.2 of Q-0225, Q-0228).** Correct the doc: there IS a central resolver, and the three named difflib anchors aren't fuzzy. Carry forward the shipped guards (destructive carve-out, ambiguity delta, min-len, loop-breaker). Make two structural fixes: (a) **derive safe/destructive from the command manifest's authority label / reversibility flag, not a hand-list** (C-1, Q-0226) so the classification can't drift; (b) **split** the rung-2 *command-typo* engine from a *generic catalog-value* resolver (fold mining/names + counting/parsing; keep the recommender/preset out — they aren't edit-distance). Add a **typo→expected-outcome golden corpus** as the calibration oracle before pinning any threshold (D-class verification, S-2). Sequence C-5 **after** C-1 (candidate set + authority come from the manifest).
**Handoff seam (session B).** The `Resolution(outcome, command)` value object (`command_resolution.py:82-85`) is the boundary: LOGIC emits AUTO/SUGGEST+command/NONE + the safe/ambiguity decision; **session B (C-5 render) owns** turning SUGGEST into the decided *private, one-tap* prompt (today it's a public message, `bot1.py:577-581`) and the AUTO note, rendering nothing for NONE, and routing the confirm-tap back through the single C-1 resolver with authority re-checked at callback time.

### Invocation rung 3 — NL intent (language -> one command)
**How now.** Rung 3 as "language -> *one command*" does **not exist today**; what ships is an NL *answer* pipeline that maps language -> a grounded prose reply, never a command dispatch. One central stage `AINaturalLanguageStage` (registered at pipeline order 70, `natural_language_stage.py:61-62`; wired unconditionally in `ai_cog.py:294-299`) owns "should the bot reply?" and emits prose or a deterministic floor (`natural_language_stage.py:991-1063`) — grepping the NL path for command invocation returns nothing. ### Transactional outbox / durable at-least-once event delivery
**How now.** The internal event path is in-process, best-effort, publish-accepted, with no persistence or replay. `emit()` awaits each subscriber under a 5 s timeout and swallows failures into in-memory counters (`disbot/core/events.py:100-137`); its only forward note is Redis *sharding*, not durability (`events.py:74-79`). Mutations commit **then** emit outside the transaction — `services/settings_mutation.py:361` stamps `committed_at`, then `:385` emits the audit event, then `:400` the domain event — so a merge=deploy restart between commit and emit loses the fact. The shared audit publisher awaits `bus.emit` in memory and logs "DB state is correct, event lost" on failure (`services/audit_events.py:74-98`); its subscriber also silently drops all global-scope mutations (`services/server_logging.py:803-805`). Level-up rewards ride the same seam: `db.add_xp` commits (`services/xp_service.py:124`), `bus.emit(EVT_LEVEL_UP)` is best-effort (`:136`), and the reward role grant is inline post-commit code (`cogs/xp/listener.py:134-135`) — crash after commit = level kept, role never granted, no replay. No outbox table exists (`disbot/migrations` grep empty), `delivery_stats` are process-lifetime in-memory (`events.py:147-154`), and no restart/crash test exists (`tests/unit/runtime/test_event_bus_delivery.py`). RS05 decided this contract (`docs/runtime_contracts.md:62-85`) and already mandates idempotent handlers "after a future Redis replay" (`:92-96`) that does not exist.
**Options considered.** (1) **Transactional outbox + relay** — write an outbox row in the mutation txn, a poller publishes + marks delivered, handlers dedup by id → true at-least-once (industry standard; fits the single-Postgres seam). (2) **Best-effort in-process** — what most Discord bots ship; durability only via gateway catch-up from Discord's durable audit log (the rebuild's P-2), which recovers nothing for internal-only facts. (3) **Durable broker w/ acks** — Redis Streams consumer groups (doubles as the sharding backend) or Celery; heavier, and a Temporal-class engine is a design-spec §10 non-goal (`:1807`).
**Recommendation.** **Centralize + decide (owner-gated).** Add `EventSpec.delivery: {best_effort, at_least_once}` to the grammar (`rebuild-design-spec-2026-07-02.md:843-855`) + a kernel `outbox` StoreSpec written in-txn, default `best_effort` (nothing regresses), and mark `audit.action_recorded` and the `xp.level_up` reward path `at_least_once` iff the owner wants restart-safe audit/rewards. This turns three ad-hoc durability decisions — money `settle_once` (G-12, FINAL-REVIEW:176), gateway `catchup_policy` (P-2, FINAL-REVIEW:218-222), internal-bus nothing — into one declared, validator-enforced standard (maps to C-x centralization + the D-3 foundation-before-consumer ordering; the rebuild currently has NO Q/C for internal-event durability).
**Handoff seam (session B).** Gate-0 prerequisite — decide the delivery tier **before** K3 (DB seam) and K4 (EventBus + catalogue) freeze, because at-least-once changes both the EventSpec grammar and the mutation-txn shape (replacing the commit-then-emit at `settings_mutation.py:361-385`); retrofitting after freeze means re-touching every mutation pipeline.

### Durable time-driven scheduler / expiry sweep with downtime catch-up
**How now.** No single seam owns persisted due-times or reconciles missed windows. Three durable families each hand-roll a loop + a `WHERE due<=now` sweep: temp roles (`RoleGrantsCog._sweep_loop` @tasks.loop(minutes=5) `role_grants_cog.py:42` → `sweep_expired` `role_grants_service.py:93` → `list_expired WHERE expires_at <= $2` `utils/db/role_grants.py:39`, exactly-once via row delete at `role_grants_service.py:129`), YouTube cache purge (`media_maintenance_cog.py:45` → `DELETE … WHERE expires_at <= now()` `utils/db/youtube_video_cache.py:70`), and the automations engine (`automation_scheduler.py:145` polls `next_run_at <= NOW()`, persists next/last-run + failure_count `utils/db/automation.py:178`). For all three, boot catch-up is an IMPLICIT side effect of "select everything already past due," never a stated contract — and the automations engine both collapses missed windows to a single fire (`automation_scheduler.py:323-330`) and is GATED OFF in prod (`AUTOMATION_SCHEDULER_ENABLED` default false `automation_scheduler.py:397`; inert spawn `bot1.py:1090`). The four one-shot timers are pure in-memory `asyncio.sleep` and are silently lost on every restart: raid-lockdown restore (`security_service.py:216-240`), proof-channel timed unlock (`proof_channel_cog.py:201`), reminders (`utility_cog.py:55`), feature-flag TTL on monotonic clock (`feature_flags.py:589`). Temp-ban/tempmute — the canonical durable-timer feature — does not exist (`docs/planning/feature-completion/units/moderation.md:21`).
**Options considered.** discord.py `ext.tasks` — in-process only; durable idiom is `before_loop` + `WHERE due<=now` (what role_grants does), no built-in catch-up. Red/Carl/Dyno/MEE6 — persist the expiry timestamp server-side and re-arm/sweep pending jobs on load so anything due during downtime fires on recovery, typically via one central scheduler table. **Recommendation.** CENTRALIZE into one new kernel durable-due-time seam (a `StoreSpec`-backed due-queue: due_at + dedup key + serializable id-args, with a single boot-reconcile that fires overdue rows exactly once). Sequence it as a foundation before consumers (S-2 / Q-0220); it is a class-7 missing-standard + class-2 forgotten-capability that no C-x/D-x currently owns — propose it as a new centralization. Owner decides collapse-vs-backfill + misfire grace, and whether automation ships on-by-default.
**Handoff seam (session B).** Build the durable due-queue StoreSpec FIRST; make DeferredActionSpec persist + re-resolve-by-id + boot-reconcile + exactly-once; use temp-ban/mute as the reference consumer that forces real durability; get an owner ruling on missed-recurring-window semantics (coalesce vs backfill + grace window) and on the automation-scheduler default-on question.
